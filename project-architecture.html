Order Processing System

This page explains the project in a simple way: who sends the request, which Java classes handle it, where business logic runs, and how data is stored.

1. Big Picture

Client

Browser, Postman, curl, or frontend sends an HTTP request.

->

Controller

Receives API request and sends it to the service layer.

OrderController
->

Service

Contains main business logic like create order, cancel order, and status update.

OrderServiceImpl
->

Database

Stores orders and order items using PostgreSQL.

orders, order_items
Simple meaning: request comes from client, controller accepts it, service processes it, repository saves or reads data from database, then response goes back to client.

3. Create Order Flow

1

Client sends order

Customer id and item list are sent to the API.

POST /api/orders
2

Controller validates request

It checks required fields like customerId, items, price, and quantity.

3

Service creates order

It calculates total amount and sets status as PENDING.

4

Repository saves data

Order and order items are inserted into PostgreSQL.

5

Response is created

Saved order is converted into OrderResponse.

6

Client receives result

API returns 201 Created with order details.

4. Main API Flows

Get One Order

Used to fetch a single order by id.

GET /api/orders/{id}
  • Controller receives id.
  • Service asks repository for order.
  • If found, API returns order details.
  • If not found, API returns 404.

Get All Orders

Used to list orders with pagination and optional status filter.

GET /api/orders?status=PENDING&page=0&size=10
  • Controller receives filter and pagination.
  • Service checks whether status filter exists.
  • Repository returns matching orders.
  • API returns paged response.

Update Status

Used to move order from one status to next valid status.

PUT /api/orders/{id}/status
  • Service checks current status.
  • Service checks requested next status.
  • If transition is valid, order is saved.
  • If invalid, API returns 400.

Cancel Order

Used to cancel an order only when it is still PENDING.

PUT /api/orders/{id}/cancel
  • If order is PENDING, it becomes CANCELLED.
  • If order is not PENDING, API returns 409 conflict.

5. Order Status Lifecycle

An order cannot jump to any status. It must follow allowed steps.

PENDING
PROCESSING
SHIPPED
DELIVERED
PENDING
CANCELLED
Easy rule: normal order moves PENDING -> PROCESSING -> SHIPPED -> DELIVERED. Cancel is allowed only from PENDING.

6. Scheduler Flow

Scheduler means background task. It runs automatically without user request.

1

Every 5 minutes

Spring runs OrderStatusScheduler.

2

Find pending orders

Service asks repository for all PENDING orders.

3

Update status

All pending orders become PROCESSING.

7. Database Design

orders table

  • id
  • customer_id
  • total_amount
  • status
  • created_at
  • updated_at

order_items table

  • id
  • product_name
  • quantity
  • price
  • order_id
One order can have many order items. That is why order_items has order_id.
